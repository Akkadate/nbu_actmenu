# NBU ActMenu - แผนงานและสถานะ Phase 1 (สำหรับผู้บริหาร)

อัปเดตล่าสุด: 2026-03-04  
กลุ่มผู้อ่าน: ผู้บริหาร / คณะกรรมการกำกับโครงการ

## 1) บทสรุปผู้บริหาร
Phase 1 ได้ส่งมอบระบบพื้นฐานพร้อมใช้งานจริง สำหรับการเช็กอินกิจกรรมผ่าน LINE เพื่อยกระดับการดูแลนักศึกษาและลดภาระงานปฏิบัติการ

ผลลัพธ์ทางธุรกิจที่เกิดขึ้นแล้ว:
- ยืนยันตัวตนนักศึกษาก่อนเข้ากิจกรรม
- ติดตามประวัติการเช็กอินรายบุคคลได้
- สื่อสารข้อมูลกิจกรรมผ่าน LINE (Rich Menu + Flex)
- ฝ่ายงานสามารถสร้าง/แก้ไขกิจกรรมได้เองผ่านหน้า Admin
- มีการควบคุมการเข้าหน้า Admin เบื้องต้น

สถานะปัจจุบัน: **ส่งมอบระบบพื้นฐานและใช้งานได้จริงแล้ว**

## 2) วัตถุประสงค์ทางธุรกิจ (Phase 1)
1. ให้ผู้ใช้งานเข้ากิจกรรมได้เฉพาะนักศึกษาที่ผ่านการยืนยันตัวตน
2. ลดกระบวนการเช็กอินแบบแมนนวล
3. ส่งข้อมูลกิจกรรมถึงนักศึกษาทันทีผ่าน LINE
4. เก็บข้อมูลเช็กอินที่ตรวจสอบย้อนหลังได้
5. ให้หน่วยงานดูแลปรับกิจกรรมได้เองโดยไม่ต้องแก้โค้ดทุกครั้ง

## 3) ขอบเขตที่ส่งมอบแล้ว

### 3.1 ฝั่งนักศึกษา
- หน้า LIFF (`/liff?activity=...`)
- ฟอร์มยืนยันตัวตน (รหัสนักศึกษา/วันเกิด/บัตรหรือพาสปอร์ต)
- ตรวจช่วงเวลากิจกรรมก่อนเช็กอิน
- หน้าสรุปผลสำเร็จและการเชื่อมไป OA chat

### 3.2 ฝั่งผู้ดูแลระบบ
- หน้าแสดงรายการกิจกรรม
- สร้างกิจกรรม / แก้ไขกิจกรรม / ปิดกิจกรรม
- หน้าเอกสารคู่มือการใช้งาน
- ระบบเข้าใช้งาน Admin ด้วย passkey + login/logout

### 3.3 ฝั่งเชื่อมต่อ LINE
- Wrapper เรียก LINE API พร้อม retry
- ตั้ง rich menu ตามกิจกรรม
- ส่ง flex message ตามกิจกรรม

### 3.4 ฝั่งข้อมูล
- `students_master`
- `line_student_links`
- `activities`
- `activity_checkins` (กันเช็กอินซ้ำรายกิจกรรม/รายผู้ใช้)

## 4) แผนเทียบผลจริง (Planned vs Actual)

| Milestone | เป้าหมายที่วางไว้ | ผลที่เกิดขึ้นจริง | สถานะ |
|---|---|---|---|
| M1: Foundation | โครงระบบ + DB + env | เสร็จแล้ว | เสร็จ |
| M2: Core APIs | verify/check/enter/admin API | เสร็จแล้ว | เสร็จ |
| M3: Admin UI | จัดการกิจกรรมแบบ CRUD | เสร็จแล้ว | เสร็จ |
| M4: LIFF UX | verify + check-in + OA handoff | เสร็จแล้ว (ปรับจูนเพิ่ม) | เสร็จ |
| M5: Reporting Baseline | เก็บข้อมูลเช็กอินสำหรับรายงาน | เสร็จแล้ว (`activity_checkins`) | เสร็จ |
| M6: Access Control | ป้องกันหน้า Admin | เสร็จแล้ว (`proxy` + cookie auth) | เสร็จ |

## 5) คุณค่าที่หน่วยงานได้รับ
1. ทำงานเร็วขึ้น: สร้าง/ปรับกิจกรรมเองได้
2. ตรวจสอบย้อนหลังได้: มีหลักฐานเช็กอินรายกิจกรรม
3. สื่อสารแม่นยำขึ้น: ส่งเมนูและข้อความตามกิจกรรมจริง
4. ลดความผิดพลาด/สวมสิทธิ์: มีขั้นยืนยันตัวตนก่อนใช้งาน

## 6) ข้อจำกัดที่ยังเหลือ (Phase 1)
1. ระบบสิทธิ์ยังเป็น passkey baseline (ยังไม่ใช่ RBAC เต็มรูปแบบ)
2. Dashboard/Export เชิงรายงานยังอยู่ในแผน Phase 2
3. บางพฤติกรรมขึ้นกับ LINE client
4. ยังไม่มี audit trail และโมดูล consent แบบเต็มรูปแบบ

## 7) ความพร้อมใช้งานเชิงปฏิบัติการ
- ขั้นตอน build/deploy ผ่าน PM2 ใช้งานได้
- Nginx reverse proxy + SSL ใช้งานได้
- มี migration/seed script สำหรับฐานข้อมูล
- มีเอกสาร handover และบริบทสำหรับทำ Phase 2 ต่อ

## 8) ความเสี่ยงและการควบคุม ณ ปัจจุบัน

| ความเสี่ยง | มาตรการควบคุม |
|---|---|
| เวอร์ชัน deploy ไม่ตรงกัน | ใช้ runbook มาตรฐาน build + reload |
| ความแตกต่างพฤติกรรม LINE | มี fallback summary + ปุ่มเข้า OA |
| การเข้าถึงหน้า admin โดยไม่พึงประสงค์ | ใช้ passkey + cookie httpOnly |
| ข้อมูลเช็กอินซ้ำ | กันซ้ำด้วย unique key `(activity_key, line_user_id)` |

## 9) ข้อเสนอเพื่ออนุมัติ Phase 2
เพื่อเพิ่มผลลัพธ์ด้านการดูแลนักศึกษา ควรอนุมัติ Phase 2 สำหรับ:
- Dashboard การเข้าร่วมกิจกรรม
- Export รายงานใช้งานจริง
- Early warning กลุ่มเสี่ยง
- RBAC + audit + governance

## 10) มุมมองทรัพยากร (ระดับสูง)
- Product owner: ฝ่ายกิจการนักศึกษา/ฝ่ายทะเบียน
- ทีมเทคนิค: Frontend + Backend + DevOps + DB
- QA/UAT: ทีมหน่วยงานผู้ใช้จริง
- รูปแบบแนะนำ: Sprint 2 สัปดาห์ + รายงานผู้บริหารรายสัปดาห์

## 11) KPI แนะนำสำหรับผู้บริหาร
1. จำนวน check-in รายสัปดาห์/รายกิจกรรม
2. อัตราเข้าร่วมกิจกรรมบังคับ
3. ระยะเวลาทำรายงานต่อรอบ
4. จำนวนกลุ่มเสี่ยงที่ระบุได้ (Phase 2)
5. จำนวน incident และเวลาแก้ไขเฉลี่ย
